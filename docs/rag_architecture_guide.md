## 1 全体像（要点）
- 成功の鍵は「データ設計と運用設計」。LLM性能だけでなく、
  - 正本の管理（Single Source of Truth）
  - チャンク化＋メタデータ
  - 検索設計（フィルタ＋ベクトル）
  - ACL・監査・差分更新
  - 評価ループ
 これらをプロダクト化することが本質。

---

## 2 データ方針（原本と成果物）
- 原本は既存の場所（SharePoint / Confluence / ファイルサーバ 等）に保持し、コネクタで参照。
- RAG用成果物は「正規化→チャンク化→JSONL（チャンク単位）」を生成して保管／投入する。
- JSONLは移植可能な中間成果物（ベクトルDBやLLMが変わっても再利用可能）。

---

## 3 推奨フォーマット
- 最推奨：Markdown（.md）＋YAML front matter（タグ・メタを容易に付与）
- 次点：HTML（既存Wiki等の取り込みに現実的。ただしノイズ除去が必要）
- 厳格運用：DITA/DocBook（XML系、運用基盤を作る覚悟がある場合）
- 最終投入形：チャンク済みのJSONL

---

## 4 チャンク化（3段階フォールバック）
優先順位（王��）：
1. 見出しあり（`#` / `<h1..>`）→ 見出し階層で分割。長すぎれば段落で分割。  
2. 見出しなし → 見出し推定（ルールベースで候補抽出 → 必要ならLLMで確定 → 階層補正）  
3. 見出し判定も不可 → 固定長チャンク（ただし優先順で切る：空行/箇条書き/句点/読点は避ける）

チャンク化のコツ：
- 目標サイズは tokens 基準（日本語は tokens 換算が安全）：例 800–1200 tokens
- オーバーラップを入れる（例 100–200 tokens）→前後の文脈欠落防止
- 手順書は「ステップ単位」、規程は「条文＋但し書き」を同一チャンクに含めるルール

---

## 5 メタデータ設計（JSONLスキーマ）
チャンクが検索・ACLの単位なので、チャンクに必要なメタは必ず持たせる。推奨は二層：`docs.jsonl`（文書単位）＋`chunks.jsonl`（チャンク単位）。

例：chunks.jsonl のフィールド（最低限）
- `chunk_id`（安定ID。再生成しても同じになる）
- `doc_id`
- `text`
- `title`
- `heading_path`（["就業規則","休暇","有給休暇"] 等）
- `source_uri`（元ファイルのURL/パス）
- `source_type`（md/pdf/html 等）
- `chunk_index`（同一 doc 内の順序）
- `tags`（部門/業務/文書種別）
- `acl`（参照権限：roles/departments）
- `effective_from` / `effective_to` / `version`
- `hash`（差分検知用）
- `chunking_method`（header/inferred/fixed）

JSONLサンプル（1行）
```json
{"chunk_id":"sha256://...","doc_id":"DOC-2026-001","title":"出張規程","heading_path":["出張規程","旅費精算"],"source_uri":"sharepoint://.../syucchou.md","source_type":"md","chunk_index":3,"text":"旅費精算の手順は...","tags":["総務","出張"],"acl":["all"],"effective_from":"2025-04-01","version":"v2","hash":"sha256://...","chunking_method":"header"}
```

安定ID設計の例：
- `chunk_id = sha256(source_uri + version + heading_path.join("/") + chunk_index)`

## 6 検索（Retrieval）設計
- ハイブリッド：ベクトル検索（意味検索） + BM25 等のキーワード検索
- フィルタ：`tags` / `version` / `acl` / `effective_from` 等で候補を絞る
- 再ランキング：上位 N 件をスコアで整列（semantic + keyword）し、LLMに供給する根拠を決定
- Window 取得：ヒットチャンクの前後も併せて取得して文脈を補う
- 返却：根拠のチャンクIDと source_uri を必ず付ける

---

## 7 回答設計（根拠提示・ハルシネーション対策）
- 回答には必ず「根拠（参照チャンクの抜粋＋リンク）」を付ける
- 自信スコア閾値以下は「回答保留」または「FAQを提示→人にエスカレーション」
- 重要領域（労務/会計/規程）は FAQ（Q→Aの正答集）を別に持ち、RAGは補助にする
- ログ：どの質問でどのチャンクを使ったか記録（改善材料）

---

## 8 運用ルール（長期安定化のための必須）
- 正本の管理：原本の「最新版定義」と承認フローを明確化
- マスタ管理（タグ・分類）の責任者を決める
- 差分更新：`hash` で変更検知 → 変更チャンクのみ再生成 → ベクトルDBを差分更新
- 監査ログ：質問、参照チャンク、応答、ユーザ（誰が何を参照したか）
- 評価指標：一次解決率、正答率、未回答率、根拠提示率
- 改善ループ：誤答ログ → 文書整備 or チャンク再設計 or FAQ追加

## 9 ディレクトリ構成（パッケージの型）
```
/sources/               # 原本の参照リスト・コネクタ設定
/cache_raw/             # 取得した原本のキャッシュ（オプション）
/normalized/            # 抽出して正規化したプレーンテキスト
/chunks/                # チャンク（JSON/JSONL）
/indexes/               # ベクトルDBの投入管理・IDマップ
/logs/                  # 取り込み・変換・投入ログ
/eval/                  # Q/A検証データ・評価結果
/config/                # チャンク方針・デフォルト設定（target_tokens 等）
```

## 10 実装フロー（PoC→改善→本番）
1. PoC：代表的な数十～百文書を投入し、UIで試験運用（ファイル投入→回答ログを即確認）  
2. 評価：誤答ログを分析し、改善対象の文書を選定（上位10件をMarkdown整備など）  
3. 拡張：コネクタ追加、ACL追加、差分更新を実装  
4. 本番：運用ルール・評価指標・改善サイクルを確立してローンチ

※最初から全社を完璧に整備しようとせず、改善ループ前提で拡張すること。

## 11 デフォルト値（実務上の目安）
- 目標 tokens / chunk：800–1200 tokens（日本語）
- overlap：100–200 tokens
- top_k retrieval：K=10 → re-rank → 上位 3–5 をLLMに渡す
- 非見出し文書の初期方針：固定長（句点優先）＋前後 window 取得
- 評価頻度：週次（PoC期）→ 月次（本番）で誤答分析

## 13 このリポジトリへの適用メモ（2026-02-27時点）

本リポジトリ（製品X Small RAG Bot）では、上記設計のうち次を実装済み。

- Retrieval: TF-IDF（char n-gram）+ スコア閾値 + 許可パスフィルタ
- 回答モード: strict / hybrid
- Window取得（Phase1）: ヒットチャンクの前後チャンク補完
- 条件付き再読（Phase2）: 手順系質問または文脈不足時に元ファイル（md/txt）を再読
- 根拠返却: sources に rank/title/source_path/score を返す

運用上の推奨：

- 既定モードは strict（社内根拠優先）
- hybrid は参考情報を使う場面に限定
- Phase2 は常時ではなく条件付きで使い、ノイズ・遅延を抑える

次の改善優先：

1. 評価セットを作り、strict/hybrid/Phase2有無で比較
2. BM25 または埋め込み検索を導入して再ランキング
3. ACL/版管理メタデータの拡充（必要部門から段階導入）
