## 2. 商用 RAG プラットフォームとのギャップ分析

### 2.1 Dify の RAG パイプライン構成

```
文書取込 → 前処理 → チャンク分割 → Embedding → ベクトルDB格納
                          ↓
クエリ → Embedding → ベクトル検索 + 全文検索 → Rerank → LLM
```

Dify が提供し、現行アプリに**不足している要素**:

| # | 機能 | Dify の実装 | 現行アプリ | 影響度 |
|---|------|------------|-----------|--------|
| G1 | **Rerank モデル** | Cohere Rerank / BGE Reranker / Jina Reranker | **✅ bge-reranker-v2-m3 実装済み** | ~~最大~~ 完了 |
| G2 | **チャンクオーバーラップ** | 設定可能 (例: 50文字) | **✅ 100文字オーバーラップ実装済み** | ~~大~~ 完了 |
| G3 | **Parent-Child チャンク** | 子チャンクでマッチ→親チャンクで回答 | Phase1 近傍補完で部分代替 | **大** |
| G4 | **ベクトル DB** | Qdrant / Weaviate / Milvus / pgvector | JSON + numpy | 中（将来） |
| G5 | **チャンクサマリ自動生成** | LLM でチャンクごとのサマリを生成・検索対象化 | なし | **大** |
| G6 | **Q&A モード (Q to Q)** | FAQ に対し質問同士でマッチ | なし | 中 |
| G7 | **トークン認識型コンテキスト管理** | モデルの context window に基づきチャンク数を自動調整 | 文字数固定 (MAX_CHUNK_CHARS=2500) | 中 |
| G8 | **多形式文書取込** | PDF, DOCX, HTML, Notion, Web ページ | .md / .txt のみ | 中（将来） |
| G9 | **前処理パイプライン** | 連続空白除去、URL/メール除去、ノイズクリーニング | なし | 小 |

### 2.2 NotebookLM の差別化要素

| # | 機能 | NotebookLM の特徴 | 現行アプリとの距離 |
|---|------|-------------------|-------------------|
| N1 | **大規模 LLM (Gemini)** | 長大コンテキスト窓 (100K+ tokens) で文書全体を理解 | gemma3:4b は約 8K tokens |
| N2 | **クロスドキュメント推論** | 複数文書をまたぐ関連性発見 | 単一文書内の Phase1/2 のみ |

| N4 | **グラウンデッド生成** | 回答根拠を引用レベルで厳密に追跡 | [Doc n] 形式で簡易参照 |

## 3. 改善ロードマップ（優先順位付き）

### Phase 1: 「精度の壁」を突破する改善（効果最大・必須）

これらは Dify 水準の精度基盤に到達するための最重要改善。

#### 1-A. Rerank モデルの導入 ★最重要

**現状の問題**: 初回検索（Bi-Encoder）は高速だが精度に限界がある。hybrid3 で 7 問失敗しているのは、初回検索のスコアリングだけでは意味的な関連性を十分にキャプチャできないため。

**改善内容**:
- 初回検索で top_k=10〜20 の候補を取得
- Cross-Encoder 型の Rerank モデルで候補を精密にスコアリング
- 上位 2〜3 件を最終コンテキストとして採用

**候補モデル（Ollama / ローカル実行可能なもの）**:

| モデル | 特徴 | 日本語対応 |
|--------|------|-----------|
| `bge-reranker-v2-m3` | bge-m3 と同系列、多言語対応 | ○ |
| `BAAI/bge-reranker-large` | Cross-Encoder、高精度 | ○ |
| `jinaai/jina-reranker-v2-base-multilingual` | 多言語、軽量 | ○ |

**実装イメージ**:
```python
# 1. 初回検索: 広めに候補を取得
candidates = retrieve_topk(payload, index_path, query, top_k=20, ...)

# 2. Rerank: Cross-Encoder で精密スコアリング
reranked = rerank_model.rank(query, [c["text"] for c in candidates])

# 3. 上位 K 件を返却
final = reranked[:final_top_k]
```

**期待効果**: P@1 を 0.912 → 0.95+ にまで到達する可能性が高い。Dify でも Rerank は「精度向上に最も寄与する」と位置付けられている。

**見積り工数**: 中（app.py に rerank ステップを追加 + モデル導入）

#### 1-C. チャンクサマリ自動生成（Dify の Summary Auto-Gen 相当）

**現状の問題**: チャンクの検索は `text` フィールドの直接マッチに依存しており、意味的に近い質問でも語彙的に異なるとヒットしにくい（パラフレーズ失敗 7 問の根本原因）。

**改善内容**:
- チャンク生成時に LLM（gemma3:4b）で各チャンクの要約を自動生成
- 要約テキストも Embedding 化して検索対象に追加
- 質問 → 要約のマッチングにより、語彙ギャップを橋渡し

**実装イメージ**:
```python
# scripts/build_chunk_summaries.py（新規）
for chunk in chunks:
    summary = ollama_generate(
        model="gemma3:4b",
        prompt=f"以下のテキストを1〜2文で要約してください:\n{chunk['text']}"
    )
    chunk["summary"] = summary
    # summary も Embedding 化して検索対象に追加
```

**期待効果**: パラフレーズ・口語クエリに対する検索ヒット率を改善。Dify ではこれを「retrievability（検索可能性）を高める」機能と位置付けている。

**見積り工数**: 中（スクリプト新規作成 + chunks.jsonl スキーマ拡張 + 検索ロジック修正）

### Phase 2: 検索品質をさらに高める改善（中期）

#### 2-A. LLM ベースのクエリ変換（HyDE / Multi-Query）

**現状の問題**: 同義語展開は 7 グループの手動辞書に限定されており、口語・パラフレーズ表現のカバレッジが不足。

**改善内容**: ユーザークエリを LLM で「検索に適した形式」に変換する。

- **HyDE (Hypothetical Document Embeddings)**: クエリから仮想的な回答文書を LLM で生成し、その回答文書の Embedding で検索する
- **Multi-Query**: 1 つの質問から複数の検索クエリを生成し、それぞれの検索結果を統合

```python
# HyDE の実装イメージ
def hyde_transform(query):
    hypothetical_doc = ollama_generate(
        model="gemma3:4b",
        prompt=f"以下の質問に対する回答を、社内製品Xマニュアルの文体で書いてください:\n{query}"
    )
    # 仮想文書の Embedding で検索
    return compute_embedding(hypothetical_doc)
```

**期待効果**: 口語「とろくさい動き」→ 仮想回答「パフォーマンスが低下している場合は…」→ 正しいチャンクにヒット。

**見積り工数**: 中（LLM 呼び出し 1 回追加、レイテンシ増加のトレードオフ）

---

#### 2-B. Parent-Child チャンク構造

**現状の問題**: Phase1 の近傍補完は「検索後」の後処理であり、検索精度自体は改善しない。子チャンク（細粒度）で精密マッチし、親チャンク（広文脈）をコンテキストに使う Dify の Parent-Child 方式が理想的。

**改善内容**:
- チャンクを 2 階層化: 親チャンク（見出しセクション全体 / 1500文字）+ 子チャンク（段落単位 / 300文字）
- 検索は子チャンクに対して実施
- マッチした子チャンクの親チャンクをコンテキストとして LLM に渡す

```
見出しセクション（親チャンク: 1500文字）
├── 段落1（子チャンク: 300文字）← 検索対象
├── 段落2（子チャンク: 300文字）← 検索対象
└── 段落3（子チャンク: 300文字）← 検索対象
```

**期待効果**: 検索精度（子チャンクの細粒度マッチ）と回答品質（親チャンクの広い文脈）を両立。

**見積り工数**: 大（chunks.jsonl スキーマ変更、検索ロジック大幅修正、Embedding 再生成）

---

#### 2-C. トークン認識型コンテキスト管理

**現状の問題**: `RAG_MAX_CHUNK_CHARS=2500` の文字数固定で、LLM の実際のトークン消費量を考慮していない。日本語はトークン効率が悪い（1 文字 ≒ 1〜3 トークン）ため、実際の利用可能コンテキストを正確に把握できない。

**改善内容**:
- `tiktoken` または Ollama の tokenizer API でトークン数を計測
- LLM のコンテキスト窓（gemma3:4b = 8192 tokens）から system prompt + history を差し引いた残りをコンテキスト予算として管理
- 予算内で最大限チャンクを詰め込む

```python
# トークン予算管理
MAX_CONTEXT_TOKENS = 8192
system_tokens = count_tokens(system_prompt)  # ~300
history_tokens = count_tokens(history)        # 可変
available = MAX_CONTEXT_TOKENS - system_tokens - history_tokens - 500  # 安全マージン

# 予算内でチャンクを追加
context_chunks = []
used_tokens = 0
for chunk in sorted_candidates:
    chunk_tokens = count_tokens(chunk["text"])
    if used_tokens + chunk_tokens > available:
        break
    context_chunks.append(chunk)
    used_tokens += chunk_tokens
```

**見積り工数**: 中（tokenizer 統合 + コンテキスト組立ロジック修正）

### Phase 3: スケーラビリティと運用基盤（長期）

#### 3-A. ベクトル DB への移行

**現状の問題**: `embeddings_bge_m3.json`（244チャンク × 1024次元）を起動時にメモリにロードしている。データ量が増えると起動時間とメモリ使用量が問題になる。

**候補**:

| DB | 特徴 | 導入容易性 |
|----|------|-----------|
| **ChromaDB** | Python ネイティブ、ファイルベース、軽量 | ★★★ 最も簡単 |
| **Qdrant** | 高性能、REST API、Docker で導入可 | ★★☆ |
| **SQLite + sqlite-vss** | SQLite 拡張、ファイルベース | ★★☆ |
| pgvector | PostgreSQL 拡張、本格運用向け | ★☆☆ |

**推奨**: 現行のローカル運用には **ChromaDB** が最適（pip install で導入、ファイルベース永続化、メタデータフィルタリング対応）。

**見積り工数**: 中（ChromaDB 統合 + 既存 JSON → ChromaDB マイグレーション）

#### 3-D. LLM のスケールアップ

**現状**: gemma3:4b（約 8K tokens コンテキスト窓）。小規模モデルのため回答品質に限界あり。

**段階的アップグレード候補**:

| モデル | サイズ | コンテキスト窓 | 日本語品質 |
|--------|--------|---------------|-----------|
| gemma3:4b | 4B | ~8K | △ |
| gemma3:12b | 12B | ~8K | ○ |
| Qwen2.5:14b | 14B | 128K | ◎ |
| Qwen2.5:32b | 32B | 128K | ◎◎ |

**推奨**: VRAM に余裕があれば `Qwen2.5:14b` への移行が効果的。128K コンテキスト窓により、トークン管理の制約も大幅に緩和される。

## 8. まとめ

現行アプリは 244 チャンクの小規模データに対して P@1=0.912 を達成しており、**基本的なハイブリッド検索としては十分に動作している**。

しかし、Dify / NotebookLM と同等の精度基盤に到達するには、以下の 3 つが必須要素となる:

1. **~~Rerank モデル~~** → **完了**: bge-reranker-v2-m3（Cross-Encoder）で初回検索候補を精密再スコアリング
2. **~~チャンクオーバーラップ~~** → **完了**: 100文字オーバーラップで分割境界の情報損失を防止
3. **チャンクサマリ / クエリ変換** — 語彙ギャップの橋渡し（次の最重要課題）

現時点で P@1=0.925 に到達しており、Dify の「Hybrid Search + Rerank」構成と同等の精度基盤が整った。次の目標は P@1 ≥ 0.97（Stage 2）であり、チャンクサマリ自動生成と HyDE/Multi-Query が鍵となる。

### 動機

- ローカル GPU がない環境でもクラウド API で利用可能にする
- より高性能なモデル（GPT-4o 等）を必要に応じて選択可能にする
- 開発・本番で異なるプロバイダを使い分ける柔軟性の確保

### 実装内容（2026-04-09 完了）

Ollama 独自API をすべて OpenAI互換API に書き換え、`data/llm_config.json` で `base_url` と `api_key` を変えるだけで切り替え可能にした。抽象クラスによるプロバイダ分離ではなく、**OpenAI互換APIへの統一**というよりシンプルなアプローチを採用。

#### 実施した変更

| カテゴリ | 変更内容 |
|----------|---------|
| **設定** | `data/llm_config.json` 新規作成。`_load_llm_config()` で読み込み。環境変数フォールバック対応 |
| **Embedding** | `_compute_embedding_scores()`: `/api/embed` → `/v1/embeddings`、レスポンス解析を `data[0].embedding` に変更 |
| **Chat** | `_call_llm()` / `_call_llm_rag()`: `/api/chat` → `/v1/chat/completions`、`choices[0].message.content` で解析 |
| **Streaming** | `_stream_llm_tokens()`: NDJSON → SSE パーサに書き換え（`data: {...}` + `[DONE]` 形式） |
| **Payload** | `"options": {"temperature": ...}` → `"temperature": ...`（トップレベル） |
| **認証** | `_llm_auth_headers()` を全HTTP呼び出しに適用（API Key 設定時のみ Bearer ヘッダー付与） |
| **エラーメッセージ** | 「Ollamaに接続できません」→「LLMサーバーに接続できません」に汎用化 |
| **フロントエンド** | `app.js` に `fetch('/api/llm-config')` 追加。モデル名デフォルトを動的取得 |
| **新規エンドポイント** | `GET /api/llm-config`: モデル名・ベースURLをフロントエンドに返却 |
