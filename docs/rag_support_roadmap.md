## 1. 目的

- 社内問い合わせに対して、根拠付きで再現可能な回答を返せる状態を作る
- 文書追加や検索ロジック変更のたびに、品質を比較できる運用にする
- サポート担当が一次回答の自動化と、例外対応の切り分けをしやすい状態にする

## 10. doc_type 混在時の対応方針

- 目的: 検索結果上位に複数の `doc_type` が混在した場合でも、一貫した回答方針を保ち、一次切り分けの精度を高める。
- 優先ルール: `troubleshooting > howto > reference > faq` の順で優先度を設ける（運用で調整可）。
- 判定アルゴリズム（実装案）:
  - 取得上位 N 件の `doc_type` ごとにスコア合計（または上位Kの合計）を計算する。
  - top_score と second_score の差分 delta を計算する。
  - delta >= 0.15: トップを採用（単一スタイルで回答）。
  - 0.05 <= delta < 0.15: トップ優先だが「参考: 別観点あり」として併記。
  - delta < 0.05: 混在と判定してハイブリッド出力（両観点を明示）に切り替える。
  - 例外: `troubleshooting` が含まれる場合は優先度を高めるが、second_score が十分高ければハイブリッドとする。
- 出力方針:
  - 単一採用時: 既存テンプレート（例: `troubleshooting` は類似事例ベース）を適用する。
  - ハイブリッド時: 「主要観点→観点Aからの推奨→観点Bからの推奨→確認事項→根拠」の順で返す。
- 可視化と運用:
  - 回答レスポンスに判定結果（primary/secondary/delta/mode）を付与してログに残す。
  - 混在頻度が高い場合はタグ・`doc_type` の見直しを実施する（運用レポートを月次で作成）。

この方針はまず暫定運用ルールとして導入し、評価セットで効果を検証したうえで閾値や優先度を調整します。
